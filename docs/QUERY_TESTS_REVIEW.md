## Готов ли код к проду?

**Кратко: нет в текущем виде.**  
Тесты можно запускать в проде (или тестовом окружении) и они выполняют проверки, но с точки зрения поддержки и развития код не готов к долгосрочному использованию в проде без доработок.

**Что мешает:**

1. Один класс на ~5k строк — тяжело ориентироваться и менять.
2. Много копипаста (валидация features, сырой JSON) — любое изменение контракта ведёт к правкам во многих местах.
3. Нет явного fail-fast при отсутствии cookie — все тесты уходят в 403, а не в понятную ошибку конфигурации.
4. Зависимость теста 71 от 67 через `lastRequestId`: при пустом списке тест 71 формально «проходит», ничего не проверяя.
5. Хрупкие проверки по точному тексту сообщений API (русский текст, формулировки бэкенда).

**Что уже хорошо для «почти прод»:**

- Есть позитивные и негативные сценарии.
- Конфигурация вынесена (Config, env, `-D`).
- Чувствительные данные маскируются в логах.
- Отчётность через Allure.

## Сильные стороны

1. **Константы и структура**  
   Эндпоинты, геометрии, типы методов вынесены в константы — удобно менять окружение и данные.

2. **Allure**  
   Использование @Epic, @Feature, @Story, @Severity, @Description даёт нормальные отчёты и группировку.

3. **Модели запросов**  
   Используются POJO (`GetObjectsByCustomerConfigRequest`, `Layer`, `PageRequest` и т.д.) вместо сырого JSON там, где это уже сделано.

4. **BaseTest**  
   Общая настройка (baseUrl, cookie, requestSpec, фильтры) в одном месте.

5. **Условный тест с другой кукой**  
   `getRequestSpecWithAutotestCookie()` + SkipException, если `cookie_autotest` не задан — корректное решение.

6. **Один порядок выполнения**  
   Priority 1–78 задаёт явную последовательность (важно при зависимости тестов друг от друга).

## Проблемы и риски

### 1. Один огромный класс (~4920 строк)

- Нарушается принцип единственной ответственности.
- Сложно искать тесты по эндпоинту/сценарию.
- Высокий риск конфликтов при работе нескольких разработчиков.

**Рекомендация:** разбить на несколько классов по доменной области, например:

- `GetObjectsByCustomerConfigTest`
- `GetObjectsByPolygonTest`
- `GetObjectsByCoordinatesTest`
- `GetAggregatedObjectsByCoordinatesTest`
- `MethodConfigsTest`
- `UserRequestHistoryTest`

Общие константы и хелперы — в базовый класс или утилитный класс (например, `QueryIntersectTestBase`).

---

### 2. Дублирование проверки структуры features

Один и тот же цикл проверки полей feature (type, layerId, geometry, coordinates и т.д.) повторяется во многих тестах.

**Рекомендация:** вынести в приватный метод:

```java
private void assertFeatureCollectionStructure(List<Map<String, Object>> features) {
    for (int i = 0; i < features.size(); i++) {
        Map<String, Object> feature = features.get(i);
        Assert.assertNotNull(feature, "feature[" + i + "] отсутствует");
        // ... остальные проверки
    }
}
```

И вызывать его после `.extract().response()` и `response.jsonPath().getList("content.features")`. При изменении контракта меняется только этот метод.

---

### 3. Сырой JSON в 22+ тестах

Использование конкатенации строк для тела запроса:

```java
String requestBody = "{\n" +
    "  \"mainLayer\": {\n" + ...
```

- Легко ошибиться (запятые, кавычки, экранирование).
- Сложно поддерживать при изменении API.

**Рекомендация:** везде, где возможно, использовать существующие модели + билдеры или фабрики тестовых данных, например:

```java
// Вместо сырого JSON
GetObjectsByCustomerConfigRequest request = GetObjectsByCustomerConfigRequest.builder()
    .customerCode(CUSTOMER_CODE)
    .mainLayer(mainLayer)
    .geometry(POLYGON_GEOMETRY)
    .methodType(METHOD_TYPE_POLYGON)
    .pageRequest(pageRequest)
    .build();
```

Для негативных кейсов (отсутствующее поле, неверный тип) — отдельные фабрики: `RequestFactory.withoutCustomerCode()`, `RequestFactory.withInvalidLayerId()` и т.п.

---

### 4. Тест 71 при отсутствии lastRequestId

Сейчас при `lastRequestId == null` тест просто выходит через `return` — в отчёте он идёт как **PASSED**, хотя ничего не проверил.

**Рекомендация:** явно помечать как пропуск:

```java
if (lastRequestId == null) {
    throw new SkipException("lastRequestId не установлен (тест 67 вернул пустой список или не выполнялся)");
}
```

Тогда в Allure/TestNG будет видно «Skipped», а не «Passed».

---

### 5. Отсутствие fail-fast при отсутствии cookie

В `setUp()` при отсутствии cookie пишется только лог, тесты всё равно запускаются и падают с 403.

**Рекомендация:** прерывать выполнение в начале:

```java
if (cookieValue == null || cookieValue.isEmpty()) {
    logger.error("Cookie не установлена...");
    throw new AssertionError("Cookie не установлена. Задайте -Dcookie=... или переменную окружения cookie.");
}
```

Либо использовать TestNG `Assume.assumeTrue(cookieValue != null && !cookieValue.isEmpty(), "Cookie не задана")` — тогда весь класс будет помечен как skipped с понятной причиной.

---

### 6. Хрупкие проверки текста сообщений об ошибках

Примеры:

- `containsString("Поле 'mainLayer' обязательно")`
- `containsString("because \"methodType\" is null")`

При смене формулировок или локализации бэкенда тесты начнут падать без реальной поломки контракта.

**Рекомендация:** по возможности проверять только:

- код ответа (400/500);
- наличие поля `message` (или аналога);
- при необходимости — ключевые слова или коды ошибок, а не полную фразу.

Точный текст лучше проверять только в отдельных «контрактных» тестах, если это требование.

---

### 7. BaseTest в src/main/java

`BaseTest` используется только в тестах, но лежит в `src/main/java`. Обычно тестовую инфраструктуру держат в `src/test/java`.

**Рекомендация:** перенести `BaseTest`, `Config`, `SensitiveDataFilter` в `src/test/java` (например, в пакет `com.dt.base`), чтобы граница «продакшен-код / тестовый код» была очевидной.

---

### 8. Нет параметризации однотипных негативных тестов

Много тестов отличаются только телом запроса и ожидаемым статусом/сообщением. Их можно объединить через `@DataProvider`:

```java
@DataProvider(name = "invalidBodies")
public Object[][] invalidBodies() {
    return new Object[][] {
        { bodyWithoutCustomerCode(), 500, "customerCode" },
        { bodyWithoutMainLayer(), 500, "mainLayer" },
        ...
    };
}

@Test(dataProvider = "invalidBodies")
public void testInvalidRequest(String body, int expectedStatus, String messageFragment) { ... }
```

Это сократит дублирование и упростит добавление новых негативных кейсов.

## Рекомендации по приоритетам

| Приоритет | Действие | Эффект |
|-----------|----------|--------|
| Высокий | Fail-fast в `setUp()` при отсутствии cookie | Понятные падения в CI вместо десятков 403 |
| Высокий | В тесте 71 при `lastRequestId == null` бросать SkipException | Честный отчёт (Skipped, а не Passed) |
| Высокий | Вынести проверку features в `assertFeatureCollectionStructure(...)` | Меньше дублирования, проще менять контракт |
| Средний | Разбить класс на 5–6 классов по эндпоинтам/фичам | Удобнее навигация и код-ревью |
| Средний | Заменить сырой JSON на модели/билдеры/фабрики | Меньше ошибок и проще поддержка |
| Средний | Смягчить проверки сообщений (не точная строка, а код + ключевые слова) | Меньше ложных падений при смене текстов API |
| Низкий | Перенести BaseTest в src/test/java | Чище структура проекта |
| Низкий | Ввести @DataProvider для однотипных негативных тестов | Меньше кода, проще расширять |

## Итог

- **Оценка: 6.5 / 10** — хорошее покрытие и базовая структура, но монолитный класс и дублирование снижают оценку.
- **К проду в текущем виде не готов** — сначала стоит внедрить fail-fast по cookie, исправить тест 71, уменьшить дублирование и по возможности разбить класс.
- После выполнения рекомендаций высокой и средней важности код можно считать готовым к использованию в проде (запуск в CI, регрессии, отчёты Allure).
